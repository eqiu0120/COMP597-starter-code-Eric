===============================================================
Measuring the Energy and Computational Efficiency of RegNet
Training Across Batch Sizes
===============================================================

Abstract
--------
We characterize the energy consumption, throughput, and GPU
utilization of training RegNet y-128GF, a large convolutional
neural network for image classification, under three batch size
configurations. Using a combination of hardware-level energy
counters (NVML), software-level carbon tracking (CodeCarbon),
and external system monitoring (nvidia-smi, psutil), we perform
fine-grained per-phase measurements across the forward pass,
backward pass, and optimizer update. Our results show that
while larger batch sizes improve GPU utilization and throughput,
all configurations exhibit periodic underutilization driven by
DataLoader epoch boundary reshuffling and memory-bandwidth-bound
optimizer steps. These idle periods represent concrete
opportunities for improving training energy efficiency.


1. WORKLOAD DESCRIPTION
========================

Model and Task.
We train RegNet y-128GF [1], a
radial-design convolutional neural network with approximately
693 million parameters (~2.6 GB of model weights), on a
standard image classification task using synthetic
ImageNet-format data. The model is implemented via torchvision
and trained with cross-entropy loss and SGD (lr=0.1,
momentum=0.9, weight_decay=1e-4) on 2,000 fake images stored
on the SLURM cluster shared filesystem (FakeImageNetSmall).
Images are loaded via torchvision's ImageFolder with a
ToTensor() transform and no augmentation. While the dataset
is synthetic, the model architecture and parameter count are
identical to a real-world deployment, making the computational
profile representative.

Hardware Setup.
All experiments were conducted on a SLURM-managed compute
node (gpu-teach-03) at McGill University under the
energy-efficiency-comp597 account (QOS: comp597). The job
was allocated 1 GPU, 4 CPU cores, and up to 16 GB of system
memory per the course QOS limits. The full node hardware is:
  - GPU:  NVIDIA RTX 5000 Ada Generation (32 GB VRAM, ~250W TDP)
  - CPU:  AMD EPYC 9474F (192 logical cores total on node;
          4 cores allocated per job)
  - RAM:  503 GB system memory
  - OS:   Linux 5.15.0, PyTorch 2.9.1+cu128

Terminology.
We adopt the following definitions consistently throughout
this report, as specified in the course guidelines:

  Step: A single parameter update of the model, consisting
  of three sequential phases:
    (1) Forward pass  — compute predictions and loss from
        one mini-batch of samples
    (2) Backward pass — compute gradients via
        backpropagation
    (3) Optimizer update — apply SGD parameter update
        using computed gradients

  Epoch: One complete pass through all 2,000 training
  samples. The number of steps per epoch equals
  2000 / batch_size (62 steps at bs=32, 125 at bs=16,
  250 at bs=8).

  Phase: Any one of the three sub-operations within a step
  (forward, backward, optimizer).

The maximum feasible batch size was determined to be 32
(OOM at batch size 64 due to GPU memory constraints with
the 3.2 GB model). We therefore evaluate batch sizes
32, 16, and 8 — three consecutive powers of two.


2. METHODOLOGY
===============

Experiment Design.
All experiments run for a fixed wall-clock budget of 300
seconds (5 minutes). Training uses a time-based loop
(TimedVisionTrainer) that processes steps continuously
until the deadline is reached, rather than training for
a fixed number of epochs. This ensures all batch size
configurations are evaluated under identical time budgets,
enabling fair cross-batch comparison. Each experiment
configuration was repeated 3 times; reported metrics
represent the mean ± standard deviation across runs.

We conducted three experiment types:

  Exp1 (noop): Training runs with no measurement
  instrumentation whatsoever. This establishes the
  baseline throughput with zero measurement overhead.
  Only the SLURM job log is recorded.

  Exp2 (codecarbon_full): A single CodeCarbon
  OfflineEmissionsTracker is started at the beginning
  of training and stopped at the end, producing one
  aggregate energy and CO2 measurement per run. This
  captures end-to-end system energy (CPU + GPU + RAM)
  with minimal instrumentation overhead.

  Exp3 (combined): Full fine-grained instrumentation.
  Per-phase timing via CUDA-synchronized wall-clock
  timestamps, per-step NVML direct GPU energy counters,
  and per-substep CodeCarbon measurements are all active
  simultaneously. External nvidia-smi logs GPU utilization,
  memory, power draw, and SM clock at 0.1-second intervals.
  A separate CPU monitor (psutil) records CPU utilization
  and RAM usage every 0.5 seconds.

Batch Size Selection.
The maximum batch size that fits in GPU memory without
out-of-memory (OOM) error was determined empirically:
batch size 32 succeeds while batch size 64 triggers a
CUDA out-of-memory error. Following course guidelines,
we evaluated batch sizes 32, 16, and 8 — three
consecutive powers of two below the maximum.

Training Configuration.
All runs use SGD with momentum (lr=0.1, momentum=0.9,
weight_decay=1e-4) and a StepLR learning rate scheduler
(step_size=100, gamma=0.1). The DataLoader is configured
with num_workers=4, shuffle=True, and no pin_memory or
prefetch_factor. These settings are held constant across
all batch sizes and experiment types to ensure fair
comparison.

Synchronization for GPU Timing.
GPU operations execute asynchronously with respect to
the CPU. To obtain accurate phase-level timestamps, we
insert explicit torch.cuda.synchronize() calls before
and after each phase boundary. This ensures the CPU
timestamp captures the true end of GPU execution rather
than the moment the CPU issued the kernel call. We
synchronize only at the three phase boundaries per step
(before/after forward, backward, and optimizer), not at
finer granularity, to minimize the idle time introduced
by synchronization itself. A fourth component,
data_loading, is measured as the gap between the end of
the optimizer step and the start of the next forward
pass, ensuring the four components sum exactly to the
total step wall-clock time.

Energy Measurement.
CodeCarbon's OfflineEmissionsTracker is configured with
measure_power_secs=0.5, sampling system power (CPU +
GPU + RAM) every 500 ms via NVML and RAPL. This
satisfies the requirement of at least 5 NVML energy
counter updates per measurement window, given that the
NVML energy counter updates every ~100 ms. Per-step
energy is inferred by prorating the 500 ms sampled
energy value across the steps that completed within
that window, weighted by step duration. Carbon intensity
uses Quebec's regional grid factor (~26 g CO2/kWh,
clean hydroelectric), applied automatically by
CodeCarbon's offline mode.

External System Monitoring.
In addition to in-process measurements, two external
monitors run as background processes launched by the
SLURM job script alongside the training process:
(1) nvidia-smi polls GPU utilization, memory usage,
power draw, and SM clock frequency every 0.1 seconds
and writes to a per-job CSV file; (2) a custom psutil-
based CPU monitor records CPU utilization and RAM usage
every 0.5 seconds using psutil.cpu_percent(), which
returns the system-wide average utilization across all
logical cores on the node (192 cores on gpu-teach-03),
not per-process or per-allocated-core utilization.
The reported values (~1%) therefore reflect total node
CPU load divided by 192 cores. To interpret in terms
of our allocated cores: 1% × 192 / 4 ≈ 48% average
utilization per allocated core, consistent with 4
DataLoader worker processes actively prefetching data.
These external logs are not subject to Python GIL
contention and provide an independent view of system
resource usage throughout the full job duration,
including startup and shutdown.

Overhead Validation.
To quantify the cost of instrumentation itself, we
compare throughput between Exp1 (noop, zero overhead)
and Exp3 (combined, full instrumentation) under
identical conditions. The target is that instrumentation
should not increase end-to-end training time by more
than ~5%. Results of this comparison are reported in
Section 3.4.

===========

3.1 End-to-End Metrics

instrumentation) averaged over 3 runs. All runs
completed in exactly 300 seconds by design.

A counterintuitive but important result is that smaller
batch sizes complete more steps (2028 vs. 845) in the
same time budget. This is because each step processes
fewer samples, making individual steps faster. However,
throughput measured in samples per second is highest
at bs=32 (90.11), because each step at bs=32 processes
32 images efficiently whereas at bs=8 each step
processes only 8 images with proportionally higher
overhead per sample.

Table 2 reports total system energy from Exp2
(end-to-end CodeCarbon), broken down by component.

Table 2: End-to-end energy consumption (Exp2, mean over 3 runs)
----------------------------------------------------------------
Batch Size  Total (kJ)  GPU (kJ)  CPU (kJ)  RAM (kJ)  CO2 (mg)
----------  ----------  --------  --------  --------  --------
32          193.05      64.66     107.73    20.66     211.14
16          191.93      63.55     107.68    20.69     209.92
8           189.88      61.61     107.64    20.62     207.68

At first glance, CPU energy appears to dominate total
consumption at ~56% (107.7 kJ) while the GPU contributes
only ~33% (63–65 kJ). However, this breakdown requires
careful interpretation. CodeCarbon measures CPU energy
via Intel RAPL, which reads hardware energy counters at
the CPU socket level — not per process or per allocated
core. On a shared SLURM node, the RAPL counter reflects
the entire socket's power consumption, including all
other jobs co-located on the same physical CPU. Our
actual CPU utilization is ~1% across 4 allocated cores
(Figure 6), meaning our workload's true CPU contribution
is negligible. The 56% CPU figure is therefore an
artifact of shared-node attribution rather than a
reflection of our job's CPU load. GPU energy (33%,
via NVML) is reliable because NVML isolates our specific
GPU device. RAM energy (11%) is estimated from memory
size and a fixed coefficient. This is a known limitation
of CodeCarbon on shared HPC clusters: per-job CPU energy
attribution via RAPL is only meaningful on dedicated
nodes. Total energy decreases only slightly across batch
sizes (~1.7% from bs=32 to bs=8), because all runs
consume the same 300s at similar average power. CO2
emissions are very low (~210 mg per run) owing to
Quebec's clean hydroelectric power grid.

Comparing Exp2 against Exp3 (combined, fine-grained)
reveals an important measurement scope difference. Exp2
reports 7–14% more total energy than Exp3 (193 kJ vs.
~180 kJ at bs=32), growing with smaller batch sizes.
This is because Exp2's CodeCarbon tracker measures the
entire job duration including CUDA initialization and
shutdown, and accounts for CPU + GPU + RAM together,
while Exp3's NVML counters measure only active GPU
energy during training steps. Neither is wrong — they
measure different scopes — but Exp2 provides the more
complete picture of total carbon cost.

3.2 Per-Phase Timing Analysis

Figure 3 shows the average duration per phase (forward,
backward, optimizer) for each batch size, measured in
Exp3 using CUDA-synchronized timestamps and averaged
over all steps across 3 runs.

[FIGURE 3: compare_phase_times.png — Three grouped bar
plots (one per batch size) showing mean ± std for
forward, backward, optimizer phase duration. Column-width.]

As shown in Figure 3, the backward pass is consistently
the most time-consuming phase across all configurations.
At bs=32, backward takes 189 ms on average versus 121 ms
for forward, 61 ms for the optimizer, and ~31 ms for
data loading. These four components sum to the measured
mean step time of ~394 ms, confirming complete phase
accountability. The backward dominance is expected:
gradient computation requires storing and reprocessing
intermediate activations from every layer, roughly
doubling the computation of the forward pass.

Phase durations scale sub-linearly with batch size.
Moving from bs=32 to bs=8 reduces forward pass time from
121 ms to 40 ms (3× reduction for 4× fewer samples),
indicating that the GPU has diminishing returns at
smaller batches where fixed kernel launch and memory
transfer overheads become relatively more significant.

The optimizer step duration (61 ms at bs=32, 61 ms at
bs=8) is notably batch-size-independent. This is because
the optimizer update reads and writes all model parameters
and gradients — whose size depends only on model
architecture, not batch size. This batch-size invariance
of the optimizer phase has important implications
discussed in Section 3.3.

A significant concern with per-phase measurement at
smaller batch sizes is NVML resolution. The NVML energy
counter updates every ~100 ms, and GPU utilization rates
represent averages over 1/6 to 1 second. At bs=8, the
forward pass completes in only 40 ms — well below the
NVML resolution window. This means per-phase NVML energy
readings at bs=8 are averages that span across phase
boundaries rather than capturing a single phase. We
account for this by using CodeCarbon's 500 ms sampling
interval to infer per-step energy, which spans multiple
NVML updates and is therefore more reliable.

3.3 GPU Utilization and System Timelines

Figure 5 shows GPU utilization, GPU memory usage, and
CPU utilization over the full 300-second training run
for each batch size, measured externally via nvidia-smi
and psutil.

[FIGURE 5: bs_32/gpu_util.png — GPU utilization timeline
for bs=32. Companion figures: bs_16/gpu_util.png,
bs_8/gpu_util.png. Arrange as 3-row subfigure panel.]

[FIGURE 6: bs_32/cpu_util.png — CPU utilization timeline.
Arrange alongside Figure 5 as a companion panel.]

[FIGURE 7: bs_32/gpu_memory.png — GPU memory timeline.]

A consistent pattern across all batch sizes is the 15-
second initialization period at the start of each run
where GPU utilization is zero. This reflects CUDA
runtime initialization and first-time kernel compilation
(JIT), as discussed in Section 3.2.

Beyond the warmup period, GPU utilization never reaches
100% for any batch size. Steady-state mean utilization
(measured from t=20s to t=305s) is:
  bs=32:  85.1% (std=8.5%,  min=46%)
  bs=16:  77.9% (std=10.9%, min=19%)
  bs=8:   77.3% (std=3.4%,  min=67%)

At bs=32, GPU utilization dips are rare (2 events) and
occur exclusively at epoch boundaries, where the
DataLoader exhausts the dataset and reshuffles the
sample index before beginning the next epoch. With 62
steps per epoch at bs=32, these boundaries occur every
~25 seconds, and are clearly visible as periodic valleys
in Figure 5. The minimum observed utilization of 46% during these
stalls represents the lowest point of a brief periodic
dip — not a sustained level — caused by the DataLoader
workers reshuffling the dataset index at epoch end
before the next batch can be supplied to the GPU.

At bs=16, the pattern becomes more complex. With steps
taking ~270ms, epoch boundaries occur every ~34s and
the optimizer step is beginning to contribute
meaningfully to idle time — resulting in 4 dip events
split evenly between the two causes. Epoch boundary
dips occur at t=35s and t=95s (near epoch boundaries
at t=34s and t=100s), while optimizer step dips occur
at t=85s and t=150s. The most severe dip reaches 19%
at t=85s — an extended optimizer stall occurring between
two back-to-back epoch boundaries where DataLoader
pressure is at its highest. The wide standard deviation of
10.9% reflects that the three runs hit these stalls
at slightly different points, so the averaged curve
shows more variability than any individual run — making
bs=16 visually appear the least stable of the three
configurations.

At bs=8, the character of underutilization shifts
fundamentally: 7 of 9 dip events are attributed to the
optimizer step rather than epoch boundaries. With each
step completing in only ~160 ms total, the optimizer
update (which takes ~61 ms, or 38% of step time) is
clearly visible as a periodic GPU stall. The optimizer
is memory-bandwidth bound — it must read all model
parameters and gradients from GPU memory and write
updated weights back — but performs minimal arithmetic.
As a result, GPU compute units sit idle during this
phase, unable to overlap with useful computation.
Paradoxically, despite more stall events at bs=8, the
standard deviation in utilization is lower (3.4%) than
at bs=32 (8.5%), because the stalls are more uniform
and predictable.

CPU utilization remains consistently low (~0.8–1.0%)
throughout all runs for all batch sizes, confirming
that our workload places negligible demand on the
CPU — DataLoader prefetching is fast because the
2,000-image dataset fits entirely in worker memory
after the first epoch, requiring no further disk I/O.
This also explains why the CodeCarbon CPU energy
reading (56%) is misleading: RAPL reads the full
socket shared with other SLURM jobs, not our
process's actual consumption. A notable drop in GPU utilization
is also visible near t=280–300s in all batch sizes,
most prominently at bs=8. This is caused by the
300-second training deadline being reached mid-epoch:
the training loop exits immediately, leaving the GPU
idle during Python cleanup and job shutdown. This
terminal dip is an artifact of the time-bounded
training design and is not accounted for in our
steady-state analysis.

3.5 Batch Size Effects — Fine-Grained Anomaly Analysis

[FIGURE 9a: bs_32/cc_energy_per_step.png — Energy per step
for bs=32. FIGURE 9b: bs_16/cc_energy_per_step.png — bs=16.
FIGURE 9c: bs_8/cc_energy_per_step.png — bs=8.
FIGURE 10a: bs_32/throughput.png — Throughput per step bs=32.
FIGURE 10b: bs_16/throughput.png — bs=16.
FIGURE 10c: bs_8/throughput.png — bs=8.
Arrange as two 3-panel rows.]

During a DataLoader reshuffle stall the GPU drops to
idle power (~50W from ~240W), so total energy consumed
during the stall period always decreases — this is the
physically correct behavior. How that decrease appears
in the per-step energy plots differs by batch size,
due to CodeCarbon's per-step attribution.

At bs=32 (Figure 9a), the dips are real and direct:
the stall falls within a single ~394ms step window,
so that step records genuinely low energy. The dips
occur consistently every ~62 steps throughout the
entire run, matching the epoch boundary period
(2000/32 = 62 steps/epoch), confirming DataLoader
reshuffles as the cause.

At bs=16 and bs=8 (Figures 9b–9c), the anomalies
appear as spikes rather than dips. This is a
per-step attribution artifact: the stall inflates
the wall-clock time of the first post-stall step
(the GPU sits idle waiting for the prefetch queue
to refill). CodeCarbon attributes that entire
inflated step duration as energy, making it appear
expensive even though the GPU was drawing ~50W for
most of it. The spike does not mean that step
consumed more energy — it is a lag indicator
signalling that a stall occurred just before it.

The patterns also differ in consistency. At bs=16,
spikes are prominent in the first ~500 steps (first
4 epochs) but fade afterwards as the 2,000-image
dataset becomes fully cached in the node's RAM,
eliminating the slow disk-read component of each
reshuffle. At bs=8, spikes are irregular and
concentrated between steps ~400–800 rather than
uniformly at every 250-step epoch boundary. This
inconsistency suggests external shared filesystem
contention on the SLURM cluster during that window
— any event draining the prefetch queue produces
the same per-step spike regardless of whether an
epoch boundary coincides.

The optimizer stall (~61ms per step) does not
produce visible energy anomalies — it is too short
relative to CodeCarbon's 500ms sampling window to
cause a measurable per-step deviation.

Throughput (Figures 10a–10c) mirrors the energy
patterns. At bs=32, periodic throughput dips align
with the energy dips at epoch boundaries. At bs=16,
dips are concentrated early then fade with cache
warmup. At bs=8, irregular dips match the same
~400–800 step window as the energy spikes.

CO2 follows energy directly given Quebec's fixed
carbon intensity. The root cause of all anomalies
is GPU idle time from DataLoader stalls, not model
computation itself.

Figure 11 shows the total accumulated energy per phase
(forward, backward, optimizer) over the full 300-second
run, measured via CodeCarbon per-substep tracking.

[FIGURE 11a: bs_32/cc_energy_substeps.png
 FIGURE 11b: bs_16/cc_energy_substeps.png
 FIGURE 11c: bs_8/cc_energy_substeps.png
 Arrange as 3-panel row for side-by-side comparison.]

At bs=32 and bs=16, the backward pass accumulates the
most total energy, consistent with it being both the
longest and most compute-intensive phase per step. At
bs=8, however, the optimizer step overtakes the backward
pass as the largest total energy consumer. This reversal
is a direct consequence of step count and phase scaling.
The optimizer update always processes all 693M model
parameters regardless of batch size — its duration is
batch-size-invariant (~61 ms per step across all
configurations). The backward pass, by contrast, scales
with batch size: at bs=8 each backward step takes roughly
~45 ms versus ~182 ms at bs=32 (4× fewer samples per
step). Meanwhile bs=8 executes ~2,025 steps versus ~844
at bs=32. The result is that total accumulated optimizer
time at bs=8 (~124s) exceeds total backward time (~91s),
flipping the energy breakdown. This makes explicit that
for small batch sizes, optimizer efficiency — not just
backward pass efficiency — is the dominant energy
concern.


4. DISCUSSION AND REFLECTION
==============================

Energy Distribution.
The GPU is the dominant energy consumer attributable
to our workload. CodeCarbon's reported CPU share (56%,
Table 2) is not a reliable per-job figure on a shared
SLURM node: RAPL reads the entire CPU socket, including
co-located jobs, and our actual CPU utilization is ~1%.
The GPU energy (33%, via NVML) is device-isolated and
accurately reflects our workload. This distinction
matters for sustainability analysis — optimizing CPU
data loading pipelines would have minimal real energy
impact here, whereas GPU efficiency improvements
(utilization, batch size, mixed precision) directly
reduce our job's actual energy cost.

Within the GPU workload, the backward pass dominates
per-step energy at bs=32 and bs=16, consuming roughly
2× the energy of the forward pass (Figure 3). At bs=8,
however, the optimizer step accumulates the most total
energy over the full run (Figure 11c) — because its
duration is batch-size-invariant while the backward
pass shrinks with smaller batches, and bs=8 runs 2.4×
more steps than bs=32 in the same 300s budget. The
optimizer step's memory-bandwidth-bound nature also
makes it the primary source of GPU underutilization
at small batch sizes, as discussed in Section 3.3.

Sources of Inefficiency.
Three categories of GPU underutilization were identified:

(1) CUDA initialization: The first 15 seconds of every
    run are wasted on kernel compilation. A single
    dummy warmup forward pass before measurement begins
    would eliminate this entirely.

(2) Epoch boundary DataLoader stalls: At bs=32, these
    account for all observed GPU idle periods. Enabling
    persistent_workers=True in the DataLoader would
    keep worker processes alive between epochs,
    eliminating the reshuffle-induced stall with a
    single configuration change.

(3) Memory-bandwidth-bound optimizer steps: At bs=8,
    the optimizer step causes 7 of 9 observed GPU idle
    events. Gradient accumulation (accumulating gradients
    over 4 small batches before each optimizer step)
    would reduce the frequency of optimizer steps by 4×,
    directly addressing this bottleneck.

Trade-offs.
The results reveal a fundamental tension between batch
size and efficiency. Larger batch sizes give higher
throughput, better GPU utilization, and lower energy
per sample, but are constrained by GPU memory (OOM
at bs=64). Smaller batch sizes complete more steps
and epochs in the same time budget but pay proportionally
more for fixed per-step overheads. For this workload,
bs=32 represents the Pareto-optimal point: it achieves
the highest throughput, best GPU utilization (85.1%),
and most favorable energy-per-sample among the tested
configurations.

Measurement Overhead.
Table 3 shows that Exp3 (combined) adds 13.5%, 21.3%,
and 25.0% per-step time overhead at bs=32, bs=16, and
bs=8 respectively — well above the ~5% target at all
batch sizes. The overhead grows at smaller batch sizes
because the instrumentation cost per step is largely
fixed: three cuda.synchronize() calls, three CodeCarbon
context manager entries/exits (each triggering NVML and
RAPL reads), and psutil/nvidia-smi polling all cost
roughly the same absolute time (~50ms per step)
regardless of how much useful computation the step
performs. At bs=32 a step is 355ms so the fixed
overhead is ~13%; at bs=8 a step is only 148ms so the
same fixed cost inflates it to ~25%. This amortization
effect explains why workloads with longer steps —
training on larger datasets or higher-resolution images
— can experience negligible overhead from the same
tooling. Exp2 introduces overhead within run-to-run
noise (~0%), confirming that end-to-end CodeCarbon
tracking is practical for production use without
perturbing the workload.

Limitations.
Several limitations of this study warrant note. First,
the 2,000-image synthetic dataset is far smaller than
real ImageNet (1.28M images). An epoch at bs=32 is
only 62 steps (~41 seconds), so epoch boundary
reshuffles occur roughly every 41 seconds throughout
the 300s run. In production training on a full dataset,
one epoch would span tens of thousands of steps —
epoch boundaries would be essentially invisible over a
300s window. The reshuffle stalls observed here are
therefore amplified artifacts of the small dataset and
should not be generalized directly to real training
workloads. Second, CodeCarbon's CPU energy measurement
via RAPL reflects the entire shared CPU socket rather
than our job's actual CPU consumption, making the
reported CPU energy share (56%) unreliable for per-job
attribution on shared SLURM nodes. GPU energy via NVML
remains trustworthy. Third, the measurements were
conducted on a shared SLURM cluster, and at least one
GPU preemption event was identified (bs=8, step #1071,
t=196s). Fourth, the NVML energy counter resolution
(~100 ms) limits per-phase energy precision at bs=8
where phase durations are 40–61 ms. Finally,
measurement overhead exceeds the 5% target at bs=16
and bs=8; future work would implement asynchronous
energy sampling to reduce the fixed probe cost.
